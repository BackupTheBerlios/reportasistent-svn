1. Overview

The AR2NL system is able to convert association rules (AR) into natural language (NL).

Let us suppose that we have got an association rule in this form (the rule being produced by the system LISp-Miner): 

Activity In a Job(he mainly sits) ∧ Activity After a Job(he mainly sits)
  ⇒0.62, 21 Education(university).

We can interpret this rule in natural language like this:

21 (i.e. 62 %) of the patients mainly sitting in their job and mainly sitting after the job also have reached university education.

Current version of the system is able to convert only rules conforming following conditions:

  • The type of quantifier is “Above average” or “Founded implication”.

  • The maximal and the minimal length of coefficient is equal 1 and

  • the type of coefficient is “Subset” or “One category”.

  • The system ignores eventual conditions in the rule.

2. Installation

Extract the distribution file LM.4ftAR2NL.zip to the home directory of LISp-Miner system.

4. Data

All linguistic data (that system uses as input data of the transformation) are saved in XML format in the directory data. You can download the templates of the data files at http://lispminer.vse.cz/download/. The meaning and the content of each particular file are described in the section Data files.

You can also download an extension for XMLMind XML Editor at http://lispminer.vse.cz/download/. This extension enables editing of the data files in the XMLMind XML Editor.

4.1. Directory structure

The structure of the data files must be created under following rules:

  1. Create subdirectory with the same name as language code according to ISO 639 (“en” for English) in the directory data.

    Important

    We later use this name of the subdirectory as a value of the parameter “Language” (see the setting of parameters described in section Rule transformation).

  2. Copy following files to the directory en (created in previous step)

    • fp.xml,

    • fpa.xml,

    • langparam.xml,

    • mn.xml and

    • mv.xml.

    Create new subdirectory in the directory en. The name of the new subdirectory should express the data domain (“what the data describe”). If we have, for example, analysed data about patients we can the directory entitle patients.

    Important

    We later use this name of the subdirectory as a value of the parameter “Domain directory” (see the setting of parameters described in section Rule transformation).

  3. Place the file entitynames.xml to the directory patients (created in previous step). In the directory patients create new subdirectory. The name of the new directory should specify the metabase that contains association rules. We tested the system on data of STULONG project [1] so we can entitle the directory stulong in our illustration.

    Important

    We later use this name of the directory as a value of the parameter “Expression directory” (see the setting of parameters described in section Rule transformation).

  4. Place the file elementary.xml to the directory stulong (created in previous step). The template of this file is generated with the program expressions.exe. We run this program from command line with one parameter:

    expressions -DSN datasourcename

    Meaning of the parameter:

    • DSN – a name of a data source that refers to the metabase with association rules.

    The program produces file elementary.xml as an output.

4.2.1. fp.xml

The file fp.xml contains formulation patterns. A formulation pattern is a base of the whole expression of an association rule in natural language. The formulation pattern is subsequently extended during the transformation.

All formulation patterns fp are contained in the root element formulationpatterns and they can contain one or more auxiliary symbols (fpa), entity links (ename), literals (literal) and optional text (optionaltext) or variants of text (choosetext).

Example 1. Sample of a formulation pattern for English

<fp id="fp2" quantifier="FUI" numofant="1+" numofsuc="1+">
  <ename symbol="Eindefart"/> <fpa symbol="Y"/>
  <optionaltext>also</optionaltext>
  <literal role="S" typeofphrase="V"/>.
  <fpa symbol="Z"/>
</fp>

4.2.2. fpa.xml

The file fpa.xml contains auxiliary symbols that are substituted in formulation patterns.

Example 2. Sample of an auxiliary symbol for English

<fpadata id="fpa4" symbol="Y" numofant="1+" numofsuc="1+">
  <ename symbol="E"/> that
  <literal role="A" typeofphrase="V" number="_"/>
</fpadata>

4.2.3. langparam.xml

The file langparam.xml contains parameters that are dependent on the language. Now, there are three parameters in the file:

Example 3. Language parameters

<!-- the conjunction "and" -->
<langparam name="conjunction" value="and"/>
  
<!-- placing of participle (posible values: "L" - left or "R" - right) -->
<langparam name="participle.position" value="R"/>
  
<!-- the symbol for decimal point -->
<langparam name="decimal.point.symbol" value="."/>

4.2.4. mn.xml

The file mn.xml contains patterns for declension of nouns and adjectives.

Example 4. Sample of a pattern in the file mn.xml

<pattern name="n1">
  <example>patient</example>
  <endsegment number="sg"></endsegment>
  <endsegment number="pl">s</endsegment>
</pattern>

4.2.5. mv.xml

the file mv.xml contains patterns for declension of verbs.

Example 5. Sample of a pattern in the file mv.xml

<pattern name="v1">
  <example>walks</example>
  <base>s</base>
  <pl></pl>
  <ptc>ing</ptc>
</pattern>

4.2.6. entitynames.xml

The file entitynames.xml contains expressions that refer to entities that are dependent on the data domain.

Example 6. Sample of an entity link

<enamedata id="ent1" symbol="E">
  <word pattern="m1" class="nominal">pacient</word>
</enamedata>

4.2.7. elementary.xml

The file elementary.xml contains elementary expressions. The program Expressions.exe produces the template of this file and the user only fill in the necessary data.

Example 7. Sample of an elementary expressions for one attribute

<expression id="ex200">
  <mbattribute>AKTPOZAM</mbattribute>
  <meaning>Activity After a Job</meaning>
  <NPinv>activity after the job</NPinv>
  <categories>
    <category>
      <value>1</value>
      <meaning>he mainly sits</meaning>
      <adjphrase relposition="R"></adjphrase>
      <nounphrase></nounphrase>
      <NAF></NAF>
      <verbphrase>mainly <word class="verb" pattern="v1t">sits</word> after
      the job</verbphrase>
    </category>
    <category>
      <value>2</value>
      <meaning>moderate activity</meaning>
      <adjphrase relposition="R">moderately active after the job</adjphrase>
      <nounphrase>moderate <NPinv>activity after the
      job</NPinv></nounphrase>
      <NAF>with <nounphrase></nounphrase></NAF>
      <verbphrase></verbphrase>
    </category>
    <category>
      <value>3</value>
      <meaning>great activity</meaning>
      <adjphrase relposition="R">highly active after the job</adjphrase>
      <nounphrase>great <NPinv>activity after the job</NPinv></nounphrase>
      <NAF>with <nounphrase></nounphrase></NAF>
      <verbphrase><word class="verb" pattern="v2">makes</word> sports
      intensively</verbphrase>
    </category>
  </categories>
</expression>

If attribute contains numeric value then the element value can contain value “all”. In this case, the element categories contains only one category – see following example. The system uses values of the category for all values of the attribute and the element <param name="N"/> is replaced with a concrete value of the attribute.

Example 8. Sample of the usage of the value “all”

<expression id="ex17">
    <mbattribute>CHLST</mbattribute>
    <meaning>Cholesterol (mg%)</meaning>
    <categories>
      <category>
        <value>all</value>
        <meaning>used for all values</meaning>
        <nounphrase>cholesterol <param name="N"></param> mmg</nounphrase>
        <NAF>with <nounphrase></nounphrase></NAF>
        <verbphrase><word class="verb" pattern="v4">has</word> cholesterol
        <param name="N"></param> mmg</verbphrase>
      </category>
    </categories>
  </expression>

5. Rule transformation

If you run the 4ftResult procedure you can browse all generated hypothesis. The 4ftResult procedure shows a list of founded hypothesis with an option to display the detail information on a hypothesis. The window with detail information (see Figure 1, “Detail of the hypothesis”) contains some bookmarks and one of the bookmarks is entitled “AR2NL”. You can see following message in the bookmark (if the system is installed correctly): 

Please run the '4ft-AR2NL' module to translate association rule to natural language.
And/or use 'Setup' to select domain, language etc.

Figure 1. Detail of the hypothesis

You have to set the necessary parameters before you can run the transformation for the first time. The button Setup displays window with parameters (see Figure 2, “Setting of the parameters”).

Figure 2. Setting of the parameters

The meaning of the parameter is following:

  • “Domain directory” – a name of the directory with domain data.

  • “Expression directory” – a name of the directory with elementary expressions.

  • “Language” – 2-letter code of the language according to ISO 639. The same 2-letter code is used as a name of the directory with linguistic data.

All parameters are saved into the metabase so you needn't set the parameters for every transformation.

Now you can run the transformation – button 4ft-AR2NL Run. The transformation takes few seconds than the system displays the result. The system can display following warning during the transformation: 

Time out expired while looking for result from 4ft-AR2NL module!

This message says that the system didn't finish the transformation of the rule during the allowed time. You can try to run the transformation once again.

The system displays by default all generated sentences including the formulation patterns that the system isn't able to convert. These “incorrect” patterns contain some elements, for instance element <literal>. We have to set the value of the parameter _show (configuration file config.xsl) to value “correct” if we want to see only correctly transformed formulation patterns.

6. AR2NL system element reference

In this section we describe every element in the DTD for the data files of the AR2NL system. The description of each element is divided into these parts: meaning of the element, allowed content, description of attributes, parents' and children's elements.

The DTD version 1.1 was released in November 2004. This document describes this current version of the DTD.

The DTD version 1.0 was released in October 2004.

6.1. Common attributes

In this section we describe the common attributes. In the description of attributes we refer to the common attributes with the string “%entity_name;”.

Attribute case contains name (or title) of a case. The value “_” means that the attribute should take the value from the attribute case of its parent element (if the parent element contains this attribute).
